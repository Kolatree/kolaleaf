## What I Want You to Verify

1. **Brief-compliance on field placement.** Email/phone verified status lives on
   `UserIdentifier.verified` + `verifiedAt` (unchanged). No `User.emailVerified` or
   `User.phoneVerified` was added. Confirm that matches your read of the brief.
2. **Additive migration.** Open `prisma/migrations/20260415113525_auth_verification_2fa/migration.sql`
   and confirm you see only CREATE / ADD / ALTER ADD. No DROP, no RENAME.
3. **Cascade correctness.** Each new token/code/challenge table cascades on user delete.
   Confirm that's what you want from a compliance standpoint (deleting a user clears all
   pending verification artifacts). Auth audit trail still lives in `AuthEvent`, which uses
   `onDelete: Restrict`, so the immutable audit log is preserved.
4. **Index coverage.** `(userId)` for lookup-by-user and `(expiresAt)` for the cleanup sweep
   that will land in 15d-15g. No composite indexes added -- the two singles should cover
   both access patterns. Flag if you'd want a composite instead.
5. **`tokenHash` uniqueness.** Email and Password reset both have `@unique` on `tokenHash`.
   `PhoneVerificationCode.codeHash` and `TwoFactorChallenge.codeHash` are **not unique** --
   short numeric codes collide frequently and uniqueness is enforced by `(userId, not consumed,
   not expired)` at the app layer. Confirm that's your read too.
6. **Legacy 2FA fields coexistence.** `User.totpSecret`, `totpEnabled`, `backupCodes` are
   still on the model. Brief said "additive only" so I did not remove them. If you want
   them removed before 15d, that's a separate (non-additive) migration and must be called
   out explicitly.
